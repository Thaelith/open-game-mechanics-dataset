# v0.3 Review Summary

This report summarizes the v0.3 preparation and review work that followed the v0.2.0 release. It is not a v0.3 release, does not mark any mechanic as reviewed, and does not change `status` or `source_confidence`.

The goal of this summary is to preserve the useful decisions from several intermediate review reports while keeping the public documentation set easier to navigate.

## Purpose

The v0.3 preparation work focused on trust, reviewability, taxonomy boundaries, relationship semantics, and practical failure-mode quality.

The main lesson is that mechanic names alone are not the dataset's value. Entries become useful when they explain prerequisites, state/input assumptions, timing and authority rules, edge cases, common bugs, and likely prototype-breaking failure modes.

## Time Rewind Architecture Gate

`time.time_rewind` should stay draft and medium-confidence until a dedicated architecture review resolves at least these areas:

- Snapshot completeness: transforms, velocity, animation state, physics state, cooldowns, health/damage state, projectiles, status effects, AI state, inventory/reward/quest state, RNG streams, and spawned/despawned object state.
- Restore order: physics restore, collision restore, carried objects, moving platforms, projectiles, enemy targets, UI state, camera state, and audio rewind behavior.
- Persistence and rewards: save/load during rewind, checkpoint behavior during rewind, reward duplication, quest flag rollback, achievement handling, and irreversible narrative choices.
- Multiplayer and authority: online co-op conflict, server authority, client prediction, rollback window separation, shared timeline rules, and reconnect behavior.
- Performance and memory: snapshot rate, buffer length, tracked actor count, object pooling, garbage collection, and streamed or unloaded chunks.
- UX clarity: what rewinds, what does not rewind, remaining history, reduced motion, audio scrub options, and failure feedback.

## Taxonomy And Boundary Criteria

The v0.3 taxonomy work clarified that the project is a practical dataset and checklist, not an objective or complete ontology of all possible mechanics.

The current criteria are documented in [taxonomy-and-categorization-criteria.md](../taxonomy-and-categorization-criteria.md). In short:

- A mechanic generally needs distinct player-facing behavior, rules/state, implementation concerns, tuning parameters, edge cases, common bugs, relationships, or design purpose.
- A variant is usually a named form of a mechanic with a similar state machine and heavily overlapping edge cases.
- A parameter is usually a tunable value, threshold, duration, cost, range, multiplier, boolean option, or mode flag.
- Entries should split when state machines, required systems, edge cases, relationships, or design purpose are meaningfully different.
- Entries should merge or reframe when the difference is only a number, label, presentation style, or duplicated checklist content.

Mechanics are context-dependent. Genre, camera, controls, level design, networking model, save/load model, progression, and surrounding systems can change the meaning and risk of a mechanic. The Mechanic Mixer is a planning aid, not proof that selected mechanics will combine cleanly.

## Locomotion Boundary Result

Walk/run/sprint were the top public boundary issue because they can look like duplicate entries or simple speed parameters.

Batch A hardened the JSON for `movement.walk`, `movement.run`, and `movement.sprint` while keeping their IDs stable:

- `movement.walk` now frames walking as baseline precision/control locomotion with ground detection, low-speed collision, input normalization, interaction alignment, prompt/dialogue/aim priority, and animation blend concerns.
- `movement.run` now frames running as default traversal locomotion with acceleration/deceleration, higher-speed collision sweeps, camera-relative input, animation blend/foot-skate risk, traversal-state transitions, surface modifiers, and prediction/reconciliation concerns.
- `movement.sprint` now frames sprinting as a high-output locomotion modifier with stamina or explicit cost, exhaustion, recovery delay, hold/toggle accessibility, camera/FOV comfort, noise/perception implications, cancellation rules, and server/client stamina authority concerns.

The locomotion entries remain separate for now because they can support distinct edge cases, common bugs, and planning concerns. They are not yet reviewed.

Current result:

- `movement.run` no longer consumes `movement.stamina_movement`.
- `movement.sprint` still consumes `movement.stamina_movement`.
- `movement.sprint` now feeds stealth/noise pressure where that context exists.
- Typed relationship count stayed at 528 after Batch A, but type counts changed because the old run stamina consume edge was replaced by a sprint noise/perception feed edge.
- All three locomotion mechanics remain draft/medium.

## Relationship Semantics And Cleanup

Relationship direction criteria are documented in [relationship-semantics-and-direction.md](../relationship-semantics-and-direction.md).

The main clarified principle is:

> A relationship should point from the mechanic being described to the other mechanic that helps explain its requirements, supports, conflicts, resource flow, upgrade path, or planning risk.

`supports` direction was the main ambiguity. The current convention is that the source supports the target. If the current mechanic benefits from another mechanic, a different type such as `requires`, `enhances`, `balances`, or a reverse `supports` edge may be clearer.

Cleanup Pass 1 addressed a small set of low-risk direction issues:

- Removed noisy or backwards support-style edges from dash/reload toward cooldown indicators when the UI-side support edge was clearer.
- Removed or reframed a too-broad reload dependency from ranged attack.
- Reversed the jump-buffering/toggle-hold support relationship so the accessibility setting supports the timing mechanic rather than the timing mechanic supporting the setting.

Graph status after Cleanup Pass 1:

- Total typed relationships: 528.
- Cleanup stayed targeted and did not change schema, site behavior, Mixer logic, mechanic status, or source confidence.
- Future relationship cleanup should prioritize reason/type/direction alignment over graph density.

The audit reinforced that future hardening should make entries answer:

- What does this mechanic need to work?
- What state does it own or depend on?
- What input, timing, resource, UI, save/load, or networking assumptions does it make?
- What breaks during reset, pause, cutscene, respawn, scene transition, save/load, or prediction?
- Why is this a mechanic rather than a variant or parameter?
